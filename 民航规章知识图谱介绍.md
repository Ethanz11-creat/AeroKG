# 民航规章知识图谱 — 完整介绍文档

> 本文档旨在帮助读者全面理解民航规章知识图谱的架构设计、抽取方法、图谱特点、存在问题以及未来应用方向，便于向他人做系统性介绍。

## 一、项目背景与目标

### 1.1 为什么做这个知识图谱？

中国民航规章体系庞杂，16部核心规章涵盖通信导航监视、空域管理、飞行规则、无人机运行等多个领域，条文之间存在大量交叉引用和条件约束关系。传统文档形式难以直观揭示这些隐含关联，导致：

- **合规检查效率低**：人工翻阅多部规章，逐条比对
- **关联关系不透明**：规章之间的引用、约束、条件依赖难以发现
- **知识难以复用**：规章知识以非结构化文本存储，无法被程序化查询和推理

**目标**：将非结构化的民航规章文本转化为结构化知识图谱，支持智能查询、合规检查和关联推理。

### 1.2 项目规模

| 指标 | 数值 |
|------|------|
| 规章文档 | 16部（CCAR系列） |
| 原始文本块 | 1,987个 |
| 最终节点 | 9,109个 |
| 最终关系 | 10,984条 |
| 全文索引 | 5个 |
| Rule追溯覆盖率 | 100% |

## 二、图谱架构详解

### 2.1 整体架构图

```
Document (文档层)
  └── CONTAINS ──→ StructuralUnit (结构层：章节条款)
                      ├── HAS_RULE ──────→ Rule (规则层：核心)
                      │                      └── DERIVED_FROM → Article (溯源层：原文)
                      ├── HAS_CONDITION ──→ Condition (条件层)
                      ├── HAS_CONSTRAINT ─→ Constraint (约束层)
                      ├── DEFINES ────────→ Definition (定义层)
                      │                      └── Term (术语层)
                      └── REFERENCES ────→ Reference (引用层)
```

### 2.2 四层架构设计

本图谱采用**四层架构**设计，从上到下依次为：

| 层级 | 节点类型 | 作用 | 设计理念 |
|------|---------|------|---------|
| **文档层** | Document | 顶层容器，一部规章对应一个节点 | 提供文档级别的聚合入口 |
| **结构层** | StructuralUnit | 章节条款的层级结构 | 保留规章原始的组织结构 |
| **知识层** | Rule / Condition / Constraint / Definition / Term / Reference | 核心知识实体 | 将规章内容拆解为可独立查询和推理的知识单元 |
| **溯源层** | Article | 规则对应的原始条款文本 | 保证每条规则都能追溯到原文，确保可解释性 |

**关键设计决策**：Condition 和 Constraint 挂载在 StructuralUnit 上（而非 Rule 上），因为一个条款中的条件/约束可能同时影响该条款下的多条规则，这是规章文本的实际语义结构决定的。

#### ② StructuralUnit — 结构单元节点（1,875个）

```
属性：id, unit_type, title, chapter, section, article_no, article_num, doc_title
```

保留规章的原始层级结构（章→节→条），unit_type 包括 article（条款）和 section（节）两种类型。每个条款节点记录了所属章节信息，便于按组织结构浏览。

#### ③ Rule — 规则节点（2,945个，核心）

```
属性：id, rule_type, modality, subject, action, object,
      normalized_subject, normalized_action, normalized_object,
      chunk_id, evidence_text, confidence
```

**这是图谱最核心的节点类型**，采用"主体-情态-动作-对象"四元组表示法：

| 字段 | 含义 | 示例 |
|------|------|------|
| subject | 规则主体 | "中国民用航空局" |
| modality | 情态词 | "应当" / "不得" / "可以" |
| action | 动作 | "统一管理" |
| object | 对象 | "全国通信导航监视工作" |
| rule_type | 规则类型 | obligation / prohibition / permission / responsibility / requirement |
| evidence_text | 原文证据 | "中国民用航空局负责统一管理全国通信导航监视工作" |

**规则类型分类**：

| rule_type | 中文含义 | 对应情态词 |
|-----------|---------|-----------|
| obligation | 义务 | 应当、必须 |
| prohibition | 禁止 | 不得、禁止 |
| permission | 许可 | 可以、仅允许 |
| responsibility | 职责 | 负责 |
| requirement | 要求 | 其他 |

**归一化字段**：normalized_subject / normalized_action / normalized_object 是对原始字段的标准化处理，包括全角转半角、别名消解（如"民航局"→"中国民用航空局"）、去除括号内缩写等。

#### ④ Condition — 条件节点（1,597个）

```
属性：id, text, condition_type, chunk_id, evidence_text, confidence
```

条件节点捕获规则生效的前提条件，分为7种类型：

| condition_type | 含义 | 示例 |
|---------------|------|------|
| scope | 适用范围 | "在中华人民共和国领域内" |
| prerequisite | 前提条件 | "具有健全的组织机构和管理制度" |
| trigger | 触发条件 | "当发生紧急情况时" |
| exception | 例外情况 | "除本规则另有规定外" |
| temporal | 时间条件 | "在规定的时限内" |
| spatial | 空间条件 | "在管制空域内" |
| operational | 运行条件 | "符合运行标准" |

#### ⑤ Constraint — 约束节点（965个）

```
属性：id, text, comparator, value, unit, chunk_id, evidence_text, confidence
```

约束节点捕获规章中的量化限制，是条件节点的数值化表达：

| 字段 | 含义 | 示例 |
|------|------|------|
| comparator | 比较运算符 | >=, <=, =, range |
| value | 数值 | 30, 1-3 |
| unit | 单位 | 天, 万元, 米, 小时 |

实际数据示例：
- "记录至少保存30天" → comparator=">=", value="30", unit="天"
- "对单位处以1万元以上3万元以下罚款" → comparator="range", value="1-3", unit="万元"

#### ⑥ Article — 溯源节点（1,166个）

```
属性：chunk_id, article_no, text, doc_title, is_virtual
```

Article 节点是后优化阶段新增的**溯源层**，核心作用是让每条 Rule 都能追溯到原始条款文本：
- **普通 Article**：从 chunks.json 同步原文文本，article_no 记录条款号
- **虚拟 Article**：为合并chunk（merged chunk）创建的虚拟节点，is_virtual=true

**设计意义**：保证了图谱的**可解释性**——用户查到一条规则后，可以立即看到它的原文出处。

#### ⑦ Definition — 定义节点（193个）

```
属性：id, term_name, definition_text, chunk_id, evidence_text, confidence
```

记录规章中的术语定义，如"通信导航监视新技术是指用于通信导航监视服务……"。

#### ⑧ Term — 术语节点（193个）

```
属性：id, name, normalized_name, category, aliases, chunk_id, evidence_text, confidence
```

术语节点与定义节点一一对应，额外提供分类和别名信息。术语分为10个类别：

| category | 含义 |
|----------|------|
| airspace | 空域类 |
| flight_rule | 飞行规则类 |
| atc_service | 空管服务类 |
| organization | 组织机构类 |
| facility | 设施类 |
| aircraft | 航空器类 |
| uav | 无人机类 |
| procedure | 程序类 |
| license | 执照类 |
| generic | 通用类 |

#### ⑨ Reference — 引用节点（159个）

```
属性：id, ref_type, ref_text, is_intra_document, normalized_ref_type, chunk_id, evidence_text, confidence
```

记录规章中的引用关系，区分**文内引用**和**外部引用**：
- 文内引用（is_intra_document=true）：引用本规章的其他条款，如"本规则第X条"
- 外部引用（is_intra_document=false）：引用其他规章，如"《民用航空电信人员执照管理规则》"

ref_type 分为 law / regulation / annex / article 四种。

### 2.4 七种关系类型详解

| 关系类型 | 起点 → 终点 | 数量 | 含义 |
|---------|-------------|------|------|
| CONTAINS | Document → StructuralUnit | 1,875 | 文档包含条款 |
| HAS_RULE | StructuralUnit → Rule | 3,138 | 条款包含规则 |
| DERIVED_FROM | Rule → Article | 2,945 | 规则派生自原文（溯源） |
| HAS_CONDITION | StructuralUnit → Condition | 1,597 | 条款包含条件 |
| HAS_CONSTRAINT | StructuralUnit → Constraint | 965 | 条款包含约束 |
| DEFINES | Term → Definition | 193 | 术语定义 |
| REFERENCES | Rule → Reference | 372 | 规则引用其他条款/规章 |

**关系方向说明**：
- REFERENCES 关系在优化后从 Rule 直接指向 Reference（原始抽取时误挂在 StructuralUnit 上，已修复）
- DERIVED_FROM 关系是后优化阶段新增的，方向为 Rule → Article，实现规则到原文的追溯

### 2.5 全文索引

为支持中文文本的高效检索，创建了5个全文索引：

| 索引名称 | 节点类型 | 索引属性 |
|---------|---------|---------|
| rule_evidence_fulltext | Rule | evidence_text |
| rule_subject_fulltext | Rule | subject |
| rule_action_fulltext | Rule | action |
| structural_unit_fulltext | StructuralUnit | title, chapter, section |
| article_fulltext | Article | text, article_no |

## 三、抽取方法详解

### 3.1 整体流程

```
原始文档 (DOCX/PDF)
    ↓ 文档解析（外部工具）
分块数据 (chunks.json, 1,987个chunk)
    ↓ [Step 1] 数据加载
    ↓ [Step 2] 文本清洗与过滤（规则驱动）
    ↓ [Step 3] LLM 知识抽取（DeepSeek-V3，核心步骤）
    ↓ [Step 4] 结果验证（规则驱动）
    ↓ [Step 5] 结果导出
节点和边文件 (kg_output/, 7,943节点, 7,927关系)
    ↓ [Step 6] 图谱优化（6步）
    ↓ [Step 7] Article节点优化（2步）
完整图谱 (9,109节点, 10,984关系)
```

### 3.2 LLM-First 架构

本项目采用 **LLM-First** 架构，核心原则是：

> **所有知识实体（subject/action/object/规则三元组/条件/约束）均由 LLM 抽取，规则方法仅用于清洗、过滤、校验和归一化。**

这与传统的"规则驱动+LLM辅助"模式有本质区别：

| 对比维度 | 规则驱动 | LLM-First（本项目） |
|---------|---------|-------------------|
| 知识抽取主体 | 正则表达式/模式匹配 | 大语言模型 |
| 规则方法作用 | 抽取知识 | 清洗/过滤/校验 |
| 灵活性 | 低（需为每种模式写规则） | 高（LLM理解语义） |
| 准确性 | 对已知模式高，未知模式低 | 整体较高，依赖模型能力 |
| 可解释性 | 高（规则透明） | 中（依赖evidence_text溯源） |

### 3.3 七步抽取Pipeline

#### Step 1：数据加载

`ChunkLoader` 从 chunks.json 加载1,987个文本块，每个chunk包含：
- `id`：唯一标识
- `text`：条款文本
- `metadata`：元数据（doc_title, article_no, chapter, section等）

#### Step 2：文本清洗与过滤

`TextCleaner` + `ChunkClassifier` 协同工作，**这一步是规则驱动的**：

**清洗操作**：
- 去除页码残留（如"— 12 —"格式）
- 修复PDF换行问题（合并断行）
- 标准化空白字符
- 检查文本长度（<15字标记为too_short，>3000字标记为review_required）
- 检测表格/附图块（包含"附表""备案表"等关键词）
- 检测附录块（"本规则自X起施行"等模式）
- 检查条号一致性

**分类过滤**：

| 分类 | 判定规则 | 处理方式 |
|------|---------|---------|
| noise | 过短/纯条号/表格/附录 | 跳过不抽取 |
| definition | 包含"是指""定义为"等 | 交给LLM抽取定义 |
| rule | 包含情态词（应当/不得等） | 交给LLM抽取规则 |
| condition | 包含条件句式 | 交给LLM抽取条件 |
| review_required | 超长文本（>3000字） | 切分后交给LLM |
| structural | 其他 | 交给LLM判断 |

**过滤结果**：1,987个chunk → 过滤108个 → 1,268个交给LLM

#### Step 3：LLM 知识抽取（核心）

**模型配置**：

| 配置项 | 值 | 设计理由 |
|--------|-----|---------|
| 模型 | DeepSeek-V3 | 中文理解能力强，性价比高 |
| 提供商 | SiliconFlow | 国内API，延迟低 |
| Temperature | 0.05 | 极低温度保证输出稳定性 |
| Max Tokens | 800 | 限制输出长度，降低成本 |
| Timeout | 45s | 防止API超时 |

**Prompt设计**：

System Prompt 定义了严格的抽取规范：
1. **绝对规则**：只输出JSON，不输出解释
2. **抽取范围**：定义类、规则类、条件类、阈值类
3. **跳过类型**：纯条号、目录、表格、生效废止历史、纯引用句
4. **输出格式**：严格的JSON模板，包含definitions/rules/conditions/constraints/references五类

User Prompt 包含文档上下文信息（chunk_id, doc_title, article_no）和原文文本。

**异步并发**：
- 使用 `AsyncKnowledgeExtractor` + `aiohttp` 实现异步并发
- 默认 batch_size=10, max_concurrency=3
- 内置MD5缓存机制，相同文本不重复调用API

**短chunk合并**：
- 长度<100字的短chunk自动合并，减少API调用次数
- 合并后的chunk_id以"merged_"开头

**长文本切分**：
- >3000字的长文本按句号/分号切分为多个子块
- 每个子块独立调用LLM抽取

**降级策略**：
- LLM调用失败时，自动降级到规则方法（`rule_based_extract`）
- 规则方法通过正则匹配情态词、条件句式、数值+单位模式来抽取
- 降级结果的confidence设为0.7（低于LLM的0.85-0.9）

**JSON解析容错**：
- 优先直接解析
- 失败则尝试去除Markdown代码块标记
- 再失败则提取{}之间的内容
- 最后尝试修复常见JSON错误（单引号→双引号、尾部逗号等）

#### Step 4：结果验证

`Validator` 执行三层验证：
1. **节点验证**：检查ID唯一性、confidence范围
2. **边验证**：检查source_id/target_id是否存在、关系类型是否合法
3. **交叉验证**：检查定义是否引用了未定义的术语、规则是否缺少action字段

#### Step 5：结果导出

`KGExporter` 将结果按节点类型分别导出为JSON文件，同时生成extraction_report.json。

#### Step 6：图谱优化（6步）

导入Neo4j后执行6步优化：

| 步骤 | 内容 | 目的 |
|------|------|------|
| Step 1 | 创建Rule全文索引 | 支持中文全文检索 |
| Step 2 | 创建Article节点层 | 为溯源做准备 |
| Step 3 | 修复REFERENCES关系方向 | StructuralUnit→Reference 改为 Rule→Reference |
| Step 4 | 标记重复规则 | 同一chunk_id下多条规则标记is_duplicate |
| Step 5 | 补充is_intra_document字段 | 区分文内引用和外部引用 |
| Step 6 | 创建DERIVED_FROM关系 | Rule→Article，实现规则溯源 |

#### Step 7：Article节点优化（2步）

| 步骤 | 内容 | 目的 |
|------|------|------|
| Step 1 | 补充Article.text | 从chunks.json同步原文文本到Article节点 |
| Step 2 | 创建虚拟Article节点 | 为merged chunk的Rule创建虚拟Article，确保100%追溯覆盖 |

## 四、图谱特点

### 4.1 核心特点

#### ① 情态驱动的规则表示

本图谱最显著的特点是采用 **"主体-情态-动作-对象"四元组** 表示规则，其中情态词（modality）是核心驱动：

- "应当" → obligation（义务型规则）
- "不得" → prohibition（禁止型规则）
- "可以" → permission（许可型规则）
- "负责" → responsibility（职责型规则）

这种表示方式直接对应法律文本的语义结构，使得规则可以被程序化地理解和推理。

#### ② 条件-约束分离设计

Condition 和 Constraint 作为独立节点类型存在，而非Rule的属性：
- **Condition**：定性条件（适用范围、前提、例外等）
- **Constraint**：定量约束（数值+单位+比较运算符）

这种分离使得条件可以被多条规则共享，约束可以被程序化地检查。

#### ③ 完整的溯源链

每条Rule通过DERIVED_FROM关系指向Article节点，Article节点保存了原始条款文本，形成了完整的溯源链：

```
Rule.evidence_text → "中国民用航空局负责统一管理全国通信导航监视工作"
    ↓ DERIVED_FROM
Article.text → 完整的原文条款
Article.article_no → "第三条"
```

这保证了图谱的**可解释性**——任何查询结果都可以追溯到原文。

#### ④ 文内/外部引用区分

Reference节点通过is_intra_document字段区分：
- 文内引用：指向同一规章的其他条款，便于分析规章内部结构
- 外部引用：指向其他规章，便于发现跨规章关联

#### ⑤ 术语归一化

通过Normalizer实现：
- 全角→半角转换
- 别名消解（"民航局"→"中国民用航空局"）
- 去除括号内缩写

使得同一实体在不同上下文中的不同表述可以被统一查询。

## 五、抽取过程中的问题与挑战

### 5.1 LLM抽取的固有问题

#### ① JSON输出不稳定

**问题**：LLM有时输出Markdown代码块包裹的JSON、包含解释文字、或格式错误。

**应对**：实现了多层JSON解析容错机制（见3.3节Step 3），但仍有个别case无法解析。实际失败率0.2%（3/1,268）。

#### ② 超长文本截断

**问题**：单次API调用限制输入2000字、输出800 token，超长条款（如列举5种违规情形的处罚条款）可能被截断。

**应对**：超长文本自动切分为子块分别抽取，但可能丢失跨句语义关联。

**实际案例**（failed_cases.jsonl中的3个失败案例）：
- chunk_00159：第一百六十条列举5种违规情形，JSON输出被截断
- chunk_00633：第十条包含12个并列职责，输出超长
- chunk_01327：第92.1011条包含3段处罚规定，结构复杂

#### ③ 情态词边界模糊

**问题**：部分条文的情态词不典型，如"由……负责""按照……执行"，LLM可能误判rule_type。

**影响**：rule_type分类可能不够精确，需要人工审核。

### 5.2 规则方法的局限

#### ① 降级抽取质量低

**问题**：规则方法（rule_based_extract）通过正则匹配抽取，只能识别简单模式，无法理解复杂语义。

**表现**：降级结果的confidence为0.7，subject/action/object的切分经常不准确。

#### ② 条件抽取粗糙

**问题**：规则方法对条件的抽取仅基于关键词匹配（"在""除""当""只有"等），无法准确识别条件的边界和类型。

### 5.3 数据质量问题

#### ① 重复规则

**问题**：同一chunk_id下可能产生多条Rule（如一条法规包含多个情态词），导致重复。

**应对**：通过is_duplicate字段标记，但未删除，保留供人工审核。

#### ② object字段缺失

**问题**：25.4%的Rule缺少object字段（749/2,945），部分是因为原文确实没有明确对象，部分是LLM未能正确抽取。

#### ③ 术语归一化不完整

**问题**：ALIAS_MAP仅包含4组别名映射（中国民用航空局/民航局等），大量术语的别名关系未被覆盖。

#### ④ 证据文本校验不严格

**问题**：evidence_text仅检查是否存在于原文中，未做精确的边界匹配，可能包含多余或不足的文本。

### 5.4 架构层面的问题

#### ① 无增量更新机制

当前每次抽取都是全量重新运行，不支持增量更新。规章修订后需要重新抽取全部内容。

#### ② 缺乏人工审核闭环

抽取结果直接导入Neo4j，没有人工审核环节。虽然有confidence和needs_review标记，但未形成审核工作流。

#### ③ 跨文档实体对齐缺失

不同规章中提到的同一实体（如"民航局"）可能被抽取为不同的Term节点，缺少跨文档的实体对齐机制。

## 六、基于图谱特点的应用方向

### 6.1 近期可实现的应用

#### ① 智能合规检查系统

**利用的图谱特点**：Rule的四元组表示 + Condition/Constraint分离 + 全文索引

**实现思路**：
1. 用户输入业务场景描述
2. 通过全文索引匹配相关Rule
3. 沿HAS_CONDITION/HAS_CONSTRAINT关系获取条件和约束
4. 自动检查是否满足所有条件和约束
5. 通过DERIVED_FROM追溯原文，生成合规报告

**示例查询**：
```cypher
// 查找"无人机"相关的所有义务型规则及其条件约束
CALL db.index.fulltext.queryNodes('rule_evidence_fulltext', '无人机')
YIELD node, score
MATCH (node)-[:DERIVED_FROM]->(a:Article)
OPTIONAL MATCH (su:StructuralUnit)-[:HAS_RULE]->(node)
OPTIONAL MATCH (su)-[:HAS_CONDITION]->(c:Condition)
OPTIONAL MATCH (su)-[:HAS_CONSTRAINT]->(cs:Constraint)
WHERE node.rule_type = 'obligation'
RETURN node.subject, node.action, node.modality,
       collect(DISTINCT c.text) as conditions,
       collect(DISTINCT cs.text) as constraints,
       a.text as original_text
```

#### ② 规章关联分析工具

**利用的图谱特点**：Reference的文内/外部引用区分 + Document层

**实现思路**：
1. 通过REFERENCES关系发现规章间的引用网络
2. 区分文内引用（规章内部结构）和外部引用（跨规章关联）
3. 可视化规章间的依赖关系图
4. 识别核心规章（被引用最多的）和孤立规章

#### ③ 主体职责图谱

**利用的图谱特点**：Rule的subject字段 + 归一化 + responsibility类型

**实现思路**：
1. 按normalized_subject聚合所有Rule
2. 区分义务(obligation)、职责(responsibility)、许可(permission)
3. 构建主体-职责-条件的三维视图
4. 发现职责交叉和空白

### 6.2 中期可探索的应用

#### ④ 规章修订影响分析

**利用的图谱特点**：REFERENCES关系 + DERIVED_FROM溯源

**实现思路**：
1. 当某条款修订时，通过DERIVED_FROM找到受影响的Rule
2. 通过REFERENCES找到引用该条款的其他规则
3. 通过Condition/Constraint关联找到依赖该条款条件的规则
4. 自动生成修订影响范围报告

#### ⑤ 约束条件自动检查

**利用的图谱特点**：Constraint的结构化表示（comparator + value + unit）

**实现思路**：
1. 将Constraint转化为可执行的条件判断
2. 用户输入具体参数值
3. 自动检查是否满足所有相关约束
4. 生成合规/不合规的详细报告

**示例**：
```
输入：无人机飞行高度 150米
检查：Constraint(comparator="<=", value="120", unit="米")
结果：不合规 — 超出限高120米
```

#### ⑥ RAG增强的规章问答

**利用的图谱特点**：全文索引 + 溯源链 + 结构化知识

**实现思路**：
1. 用户自然语言提问
2. 图谱检索相关Rule + Condition + Constraint
3. 通过DERIVED_FROM获取原文
4. 将结构化知识+原文作为上下文，送入LLM生成回答
5. 回答附带原文引用，确保可验证

### 6.3 远期可展望的应用

#### ⑦ 规章知识推理

**利用的图谱特点**：四元组规则表示 + 条件依赖关系

**实现思路**：
- 基于规则的情态逻辑推理（义务→禁止的矛盾检测）
- 条件链推理（A条件触发B规则，B规则产生C条件）
- 跨规章的规则冲突检测

#### ⑧ 规章自动摘要与对比

**利用的图谱特点**：结构化知识表示

**实现思路**：
- 按主题自动聚合相关规则，生成摘要
- 对比不同版本规章的差异（新增/修改/删除的规则）
- 对比不同规章对同一主题的规定差异

#### ⑨ 合规知识推荐

**利用的图谱特点**：规则-条件-约束的关联网络

**实现思路**：
- 根据用户角色（飞行员/空管/运营人）推荐相关规则
- 根据操作场景推荐需要满足的条件和约束
- 根据历史查询推荐关联规则

## 九、总结

民航规章知识图谱是一个**面向垂直领域的结构化知识系统**，其核心价值在于：

1. **将非结构化规章文本转化为可查询、可推理的知识网络**
2. **采用LLM-First架构**，以大语言模型为核心抽取引擎，规则方法辅助清洗和校验
3. **四层架构设计**（文档层→结构层→知识层→溯源层），兼顾结构化查询和可解释性
4. **情态驱动的规则表示**，直接对应法律文本的语义结构
5. **完整的溯源链**，每条规则都可追溯到原文条款

当前图谱已覆盖16部核心民航规章，包含9,109个节点和10,984条关系，具备全文检索、规则追溯、关联查询等基础能力。未来可在合规检查、影响分析、智能问答等方向深入应用。
